# Кейс-задача №5

Аналитический обзор кейса №3 по критериям качества.

## Анализ

Схема в целом нормальная для учебного проекта. Покрывает основные сущности: страны, города, туры, услуги, заказы. Всё связано через FK, есть индексы — с производительностью на небольших объёмах проблем не будет.

Что можно улучшить:

Во-первых, сейчас заказ привязан только к одной услуге. В реальности клиент может заказать тур + страховку + трансфер + экскурсию. Нужна промежуточная таблица order_services для связи многие-ко-многим.

Во-вторых, нет таблицы клиентов. Сейчас customer_name и phone лежат прямо в orders. Если один клиент делает несколько заказов — данные дублируются. Лучше вынести в отдельную таблицу customers.

В-третьих, нет статусов заказа. Непонятно, оплачен он или нет, подтверждён или отменён. Для реальной системы это критично.

По безопасности — на уровне схемы всё ок (FK, CHECK), но нужно ещё настраивать права доступа в СУБД, это уже за рамками DDL.

По переносимости — схема написана для MySQL, но легко адаптируется под PostgreSQL или MS SQL (что и сделано в кейсе 4). Специфичных конструкций нет.

В общем, для учебной задачи — достаточно. Для продакшена — нужно дорабатывать.
